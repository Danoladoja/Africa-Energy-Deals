## SPECIAL CONCERN — existing rows in the current Replit DB

The current DB already has `submitted_by_contributor_id` and `community_submission_id` as bare INTEGER columns (no FK). Adding a `REFERENCES` constraint to an existing column on an existing DB is done via `ALTER TABLE ... ADD CONSTRAINT ...`, not by re-running `ADD COLUMN`. The migration needs to handle both cases:

- **Fresh DB:** column created with `REFERENCES` inline.
- **Existing DB:** column already exists without `REFERENCES`; add a separate `ADD CONSTRAINT IF NOT EXISTS` (or check via `pg_constraint` lookup — Postgres does not support `IF NOT EXISTS` on `ADD CONSTRAINT` directly).

Handle this with a conditional DO block:

```sql
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_constraint
    WHERE conname = 'energy_projects_submitted_by_contributor_id_fkey'
  ) THEN
    ALTER TABLE energy_projects
      ADD CONSTRAINT energy_projects_submitted_by_contributor_id_fkey
      FOREIGN KEY (submitted_by_contributor_id) REFERENCES contributors(id);
  END IF;
END $$;
```

Same pattern for `community_submission_id`. Before running the DO block, verify that every existing value in the columns references a real row in the parent table — if any row has an orphaned FK, the constraint addition will fail. Add a pre-check query and log the orphan count; if non-zero, do **not** add the constraint, just log a warning and proceed. Do not silently delete orphaned rows.
