## DIAGNOSIS — TWO CRITICAL BUGS

### Bug A: Email dispatch uses batch endpoint that returns 403

The Resend API Logs confirm:
- `POST /emails/batch` → **403 Forbidden** (every newsletter send attempt)
- `POST /emails` → **200 OK** (all magic-link sign-in emails work fine)

The newsletter email dispatch code uses `resend.emails.batch.send()` (which calls `/emails/batch`) to send to all subscribers at once. **Your Resend plan does not include batch sending** — it returns 403. The code then **swallows the error** and marks the newsletter as "sent" with recipientCount: 13, even though zero emails were delivered.

**Proof from Resend Logs page:** The only successfully delivered emails in the last 15 days are "Sign in to AfriEnergy Tracker" magic links. Zero newsletter emails appear in the Emails tab.

### Bug B: (Now fixed) Database migration was missing

The `type` column was added to code but not migrated in PostgreSQL. This has been fixed — newsletters now save correctly. Editions 3 and 5 exist in the DB with status "sent", but no emails were actually delivered due to Bug A.

## FIX 1: Replace Batch Email Sending with Individual Sends (CRITICAL — #1 PRIORITY)

**Problem:** The email dispatch code uses `resend.emails.batch.send()` (Resend's `/emails/batch` endpoint) to send newsletters to all subscribers. This endpoint returns **403 Forbidden** on the current Resend plan. The code then swallows the error and falsely marks the newsletter as "sent".

**File:** `artifacts/api-server/src/services/email-dispatch.ts`

**What to do:**

**A. Replace batch send with a loop of individual sends:**

Find the function that sends the newsletter email (likely `sendNewsletter()` or `dispatchNewsletter()`). It currently does something like:

```javascript
// BROKEN — returns 403
const result = await resend.emails.batch.send(
  subscribers.map(sub => ({
    from: 'noreply@afrienergytracker.io',
    to: sub.email,
    subject: newsletter.title,
    html: newsletter.contentHtml,
  }))
);
```

Replace it with individual sends:

```javascript
// WORKING — uses /emails endpoint which returns 200
const results = [];
const errors = [];

for (const subscriber of subscribers) {
  try {
    const result = await resend.emails.send({
      from: 'AfriEnergy Tracker <noreply@afrienergytracker.io>',
      to: subscriber.email,
      subject: newsletter.title,
      html: newsletter.contentHtml,
    });
    results.push({ email: subscriber.email, id: result.id, status: 'sent' });
  } catch (error) {
    console.error(`Failed to send to ${subscriber.email}:`, error);
    errors.push({ email: subscriber.email, error: error.message });
  }
}

console.log(`Newsletter dispatch: ${results.length} sent, ${errors.length} failed`);
```

**B. Fix the error handling — do NOT mark as "sent" unless emails actually delivered:**

Find where the code updates the newsletter status to "sent" and recipientCount. Currently it probably does:

```javascript
// BAD — marks sent regardless of Resend response
await db.query('UPDATE newsletters SET status = $1, sent_at = NOW(), recipient_count = $2 WHERE id = $3',
  ['sent', subscribers.length, newsletterId]);
```

Change it to only mark sent if at least some emails succeeded:

```javascript
const successCount = results.length;
const status = successCount > 0 ? 'sent' : 'failed';

await db.query(
  'UPDATE newsletters SET status = $1, sent_at = $2, recipient_count = $3 WHERE id = $4',
  [status, successCount > 0 ? new Date() : null, successCount, newsletterId]
);

if (errors.length > 0) {
  console.error(`Newsletter ${newsletterId}: ${errors.length} delivery failures:`, errors);
}
```

**C. Add a small delay between sends to avoid rate limiting:**

```javascript
// Add 100ms delay between sends to avoid hitting Resend rate limits
await new Promise(resolve => setTimeout(resolve, 100));
```

**Verification:** After this fix:
1. Generate & Send a newsletter from admin
2. Check Resend Emails page — you should see individual newsletter emails appearing
3. Check your inbox — the newsletter should arrive

## Summary of Root Causes

### Critical: Email delivery completely broken
- The email dispatch code uses `resend.emails.batch.send()` → calls `/emails/batch`
- Resend free/starter plan returns **403 Forbidden** for batch endpoint
- The code **swallows the 403 error** and marks newsletters as "sent" anyway
- **Result:** Zero newsletter emails have ever been delivered. All "sent" newsletters in the DB are false positives.
- **Fix:** Switch to `resend.emails.send()` in a loop (the same endpoint that works for magic-link emails)

### Previously fixed: Database migration was missing
- The `type` column was added to code but not migrated — now resolved
- Newsletters can now be generated and saved correctly

### UI gaps: Admin panel not updated for two-product system
- No type selector for Insights vs Brief
- Copy still says "weekly briefings"
- No error states shown when generation/dispatch fails
