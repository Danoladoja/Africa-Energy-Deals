You've implemented fuzzy matching, direct API dedup, reviewer duplicate warnings, cross-channel URL linking, and the pg_trgm GIN index. Four gaps remain. Implement all four:
1. Existing Duplicate Scanner (admin tool)
Build GET /api/admin/duplicates that scans the projects table and returns groups of likely duplicates. Use the pg_trgm similarity() function you've already set up — compare project_name values within the same country, flagging any pair with similarity > 0.6. Return results as groups sorted by similarity score descending. Each result should include both projects' IDs, names, countries, developers, capacities, and the similarity score. Add a "Duplicate Scanner" section accessible from the admin dashboard that displays these groups in a reviewable list. Include pagination since results could be large on first run.
2. Duplicate Merge Workflow
Add POST /api/admin/projects/merge accepting { keepId, removeId }. It should: (a) for every column on the removeId record that is non-null, copy it into the keepId record ONLY if that column is currently null on keepId (fill gaps, never overwrite existing data), (b) update any foreign key references across all related tables (reviews, news items, pending submissions, etc.) that point to removeId to point to keepId instead, (c) delete the removeId record, (d) log the merge with both IDs and a timestamp for audit purposes. Wire a "Merge" button into the Duplicate Scanner UI from step 1 — for each duplicate pair, let the reviewer pick which record to keep, then trigger the merge. After merge, remove that pair from the displayed list.
3. Pre-Submit Similarity Check for Community Contributions
Add a public (but rate-limited) endpoint GET /api/projects/similar?name=X&country=Y that returns the top 3 existing projects with the highest name similarity in that country (using the pg_trgm similarity function). In the "Submit a Deal" form on the frontend, after the user fills in the project name and country fields, fire a debounced call to this endpoint. If any result scores above 0.5, display a "Similar existing projects" card showing the matches with their names, countries, and developers. Give the submitter two clear options: "This is the same project" (which cancels the submission and optionally links them to the existing project) or "Mine is different — continue submitting." This should not block the form — it's advisory, not a gate.
4. Scraper Name Normalization Layer
In the scraper's project matching logic (the writeCandidate helper where you just added fuzzy matching), add a normalization step that runs BEFORE the similarity comparison. The normalizer should: lowercase the name, strip common suffixes and filler words ("project", "phase 1", "phase 2", "phase 3", "phase i", "phase ii", "development", "initiative", "programme", "program"), collapse multiple spaces into one, trim leading/trailing whitespace, and remove special characters except hyphens. Apply this normalization to both the incoming scraped name AND the existing database names during the similarity check. Store the normalized form in a new project_name_normalized column (populated via a one-time migration for existing records and auto-set on insert/update going forward) so you can index it and avoid recomputing on every query. Add a GIN trigram index on this normalized column as well.
All new endpoints in steps 1 and 2 must be behind admin auth. Step 3's endpoint should be public but rate-limited to 10 requests per minute per IP. Run a one-time migration script for step 4's normalized column after deploying.
