## DIAGNOSIS

The newsletter overhaul added a `type` column to the newsletter code (INSERT and SELECT queries) but **never ran the database migration** on the PostgreSQL database. This causes:

1. **`GET /api/newsletters` → 500** — the SELECT query references a `type` column that doesn't exist
2. **`GET /api/newsletters/latest` → 500** — same issue
3. **`POST /api/admin/newsletter/preview` → job starts, AI generates content, then CRASHES on INSERT** because the `type` column doesn't exist in the actual table
4. **Admin shows "Editions (0)"** — because the list query fails silently
5. **Admin has no way to choose between "Insights" (monthly) and "Brief" (biweekly)** — only a single generic "Preview Edition" / "Generate & Send" exists

### Proof — Actual Error from Job Status

```
Failed query:
    INSERT INTO newsletters
      (edition_number, title, content, executive_summary,
       spotlight_sector, spotlight_country,
       projects_analyzed, total_investment_covered, status, type)
    VALUES ($1, $2, $3, $4, $5, $6, $7, $8, 'draft', $9)
    RETURNING id
```

The query includes `type` in the column list, but the PostgreSQL `newsletters` table has no such column.

## FIX 1: Run the Database Migration (CRITICAL — Do This First)

**Problem:** The `type` column exists in the Drizzle schema code but not in the actual PostgreSQL table.

**What to do:**

Add a migration that runs on server startup (or use `drizzle-kit push` if the project uses it). The safest approach is to add a raw SQL migration that runs before any newsletter queries.

**Option A — Use Drizzle Kit:**
```bash
npx drizzle-kit push
```
This will sync the schema to the database.

**Option B — Manual migration in server startup:**

In the server startup file (e.g., `artifacts/api-server/src/app.ts` or wherever the DB connection is initialized), add:

```sql
ALTER TABLE newsletters ADD COLUMN IF NOT EXISTS type VARCHAR(20) DEFAULT 'insights';
```

Run this migration BEFORE any route handlers fire. Use `pool.query()` or `db.execute()` to run it.

**Option C — Create a migration file:**

If the project uses a migrations folder, add a new migration:

```sql
-- Add type column for newsletter/brief distinction
ALTER TABLE newsletters ADD COLUMN IF NOT EXISTS type VARCHAR(20) DEFAULT 'insights';

-- Backfill existing newsletters as 'insights' type
UPDATE newsletters SET type = 'insights' WHERE type IS NULL;
```

**Verification:** After migration, `GET /api/newsletters` should return 200 (not 500).
