## Overview: Unified AI Chatbot Replacing Ask AI

### What Changes
The current "Ask AI" feature (⌘K shortcut, component: `src/components/ai-assistant.tsx`, search bar: `src/components/nlq-search-bar.tsx`) is a **natural language search/filter tool**. It opens as a slide-out right panel and lets users type queries like "Solar deals in West Africa above $100M" to search/filter the project database. It only returns matching deal records — it cannot generate analysis, commentary, or insights.

**We are replacing this with a unified AI chatbot** that can do BOTH:

1. **Search & lookup** (what Ask AI does today) — "Show me solar deals in Egypt above $200M"
2. **Intelligence & analysis** (new capability) — "What are the key investment trends in West African renewables?" / "Compare DFI involvement across regions" / "Give me a risk assessment for the Nigerian energy pipeline"

The chatbot will live at `/insights` as a dedicated page in the side navigation AND remain accessible via ⌘K as a slide-out panel (same chatbot, two entry points).

### What Gets Removed
- The old `ai-assistant.tsx` component (slide-out search panel) — replaced by the new chatbot
- The old `nlq-search-bar.tsx` component — replaced by the chatbot's input field
- The old Ask AI button in the top bar — replaced by a ⌘K shortcut that opens the chatbot slide-out

### What Gets Created
- A new `/insights` page with the full chatbot experience + guided insight templates
- A new chatbot slide-out component (accessible via ⌘K from any page)
- A new `POST /api/chat` backend endpoint that handles both search queries and analysis requests
- A new sidebar nav item: "Insights" with Lightbulb icon and "AI" badge

## ⛔ CRITICAL: Anti-Hallucination & Data Integrity Framework

**This is the single most important requirement of the entire build.** The AI Insights agent must NEVER fabricate, invent, or extrapolate data that does not exist in the database or in verifiable external sources. Every number, project name, country, investor, dollar amount, and statistic cited in any AI response MUST trace back to an actual record in the PostgreSQL database or a verifiable scraped external source.

### The Three-Layer Anti-Hallucination System

This must be implemented across ALL three layers — the system prompt, the backend, and the frontend. All three are mandatory.

#### Layer 1: System Prompt Constraints (instructions to Claude)

The system prompt for BOTH the chatbot and the newsletter generator MUST include these non-negotiable rules (see full prompts below — these rules are already embedded there, but they are repeated here for emphasis):

```
ABSOLUTE DATA INTEGRITY RULES — THESE OVERRIDE ALL OTHER INSTRUCTIONS:

1. NEVER invent project names, deal sizes, investor names, country data, or any statistics.
   Every data point you cite MUST come from the DATA PROVIDED below.

2. NEVER extrapolate or project numbers beyond what the data shows.
   If 69 solar projects exist in the data, say "69 solar projects" — not "approximately 70"
   or "nearly 100" or "a growing number."

3. NEVER state trends unless the data contains time-series evidence.
   Do NOT say "investment is increasing" unless you can cite specific year-over-year numbers
   from the announcedYear/closedYear fields. If the data doesn't show a clear trend, say
   "the data does not show a clear directional trend" or "insufficient time-series data to
   determine trend direction."

4. NEVER fill gaps with assumptions.
   If a field is null/empty for a project (e.g., no financingType, no capacityMw), do NOT
   guess. Say "financing type not disclosed" or "capacity data not available for X projects."

5. ALWAYS prefix external intelligence with its source.
   Say "According to IRENA's 2025 report..." or "BloombergNEF data indicates..."
   NEVER present external claims as your own analysis.

6. ALWAYS disclose data limitations.
   Every response must include at least one data caveat, such as:
   - "Based on [N] tracked projects; actual market activity may be broader"
   - "Deal sizes are self-reported and may not reflect final committed capital"
   - "[N] projects have incomplete financing data"

7. When asked about something NOT in the data, say so explicitly.
   "The tracker does not currently contain data on [topic]. I can only analyze
   what's in the database."

8. NEVER use phrases that imply certainty beyond what data supports:
   ❌ "This clearly shows..." / "It is evident that..." / "Undoubtedly..."
   ✅ "The data suggests..." / "Based on the [N] projects tracked..." / "The available data indicates..."
```

#### Layer 2: Backend Data Injection & Validation

The backend (`POST /api/chat` and the newsletter generator) MUST implement these safeguards:

**A. Inject ONLY real data — never let Claude generate from memory:**

```javascript
// CORRECT: Query the database, inject the results into the prompt
const projects = await db.query('SELECT * FROM projects WHERE technology = $1', [sector]);
const systemMessage = `
  DATA PROVIDED (${projects.length} projects):
  ${JSON.stringify(projects)}

  IMPORTANT: You may ONLY reference projects and data points from the DATA PROVIDED above.
  Do NOT use your training data to supplement with projects, statistics, or deals not listed above.
`;

// WRONG: Asking Claude to "analyze African energy markets" without injecting data
```

**B. Attach a data fingerprint to every response:**

After the Claude response is generated, the backend MUST compute and attach a `dataSummary` object that is generated from the ACTUAL DATABASE QUERY (not from parsing Claude's response):

```javascript
// Compute from the actual query results, NOT from Claude's text
const dataSummary = {
  projectsInQuery: projects.length,
  totalInvestmentInQuery: projects.reduce((sum, p) => sum + (p.dealSizeUsdMn || 0), 0),
  countriesInQuery: [...new Set(projects.map(p => p.country))].length,
  sectorsInQuery: [...new Set(projects.map(p => p.technology))].length,
  queryTimestamp: new Date().toISOString(),
  dataSource: 'afrienergytracker_postgresql'
};

// Attach to the response so the frontend can display it
response.dataSummary = dataSummary;
```

**C. Post-generation spot-check (optional but recommended):**

After Claude generates a response, run a lightweight validation pass:

```javascript
async function validateResponse(responseText, queryResults) {
  // Extract numbers mentioned in the response
  const mentionedAmounts = responseText.match(/\$[\d,.]+[BMK]/g) || [];
  const mentionedCounts = responseText.match(/(\d+)\s+projects?/g) || [];

  // Check if mentioned project count matches query
  const actualCount = queryResults.length;
  const mentionedCount = mentionedCounts[0] ? parseInt(mentionedCounts[0]) : null;

  if (mentionedCount && Math.abs(mentionedCount - actualCount) > 2) {
    // Flag potential hallucination — the AI mentioned a project count
    // that doesn't match the actual data
    console.warn(`[DATA INTEGRITY WARNING] AI mentioned ${mentionedCount} projects, actual query returned ${actualCount}`);
    // Optionally: append a disclaimer to the response
  }

  return { valid: true, warnings: [] };
}
```

#### Layer 3: Frontend Data Provenance Display

Every AI response in the chatbot and every newsletter MUST display a **data provenance badge** that is generated from the backend's `dataSummary` (NOT from the AI's text):

```
┌─────────────────────────────────────────────────┐
│ 📊 Data Source: AfriEnergy Tracker PostgreSQL    │
│ 📋 Projects analyzed: 156 │ 💰 $115.4B tracked  │
│ 🌍 40 countries │ ⚡ 9 sectors                   │
│ 🕐 Query: 2026-03-29T14:23:00Z                  │
│ ⚠️ AI-generated analysis — verify critical data  │
└─────────────────────────────────────────────────┘
```

This badge:
- Is computed from the ACTUAL database query, not from parsing the AI's text
- Shows the real timestamp of when the data was queried
- Includes a permanent disclaimer: "AI-generated analysis — verify critical data"
- Cannot be hidden or dismissed

Additionally, the frontend should render a small ⚠️ icon next to any response, with a tooltip: "This analysis is AI-generated from database records. While grounded in real data, AI interpretation may contain errors. Always verify critical figures against the source data."

5. Construct the Claude API call with:

// Pseudocode for the web intelligence scraper

interface IntelligenceSource {
  name: string;
  url: string;
  category: 'dfi' | 'industry' | 'news' | 'thinktank';
  scrapeMethod: 'rss' | 'api' | 'html';
}

interface ScrapedIntelligence {
  source: string;
  title: string;
  summary: string;       // 2-3 sentence summary
  publishDate: string;
  url: string;
  relevanceScore: number; // 0-1, how relevant to African energy
  category: string;
}

async function gatherExternalIntelligence(): Promise<ScrapedIntelligence[]> {
  // 1. Iterate through configured sources
  // 2. For each source, fetch via RSS feed, API, or HTML scraping
  // 3. Extract article titles, summaries, and dates from the last 7-30 days
  // 4. Use Claude to score relevance (0-1) and extract key data points
  // 5. Return the top 15-20 most relevant items
  // 6. Cache results for 24 hours to avoid redundant scraping
}
```

**Scraping Strategy (in priority order):**
1. **RSS feeds** — Preferred. Most sources have RSS feeds. Use a library like `rss-parser`.
2. **Public APIs** — Some sources (World Bank, IRENA) have open data APIs.
3. **HTML scraping** — Last resort. Use `cheerio` for basic HTML parsing of article listings. Only scrape publicly accessible content (article titles and summaries from listing pages — never full article text behind paywalls).

**Important:** Respect robots.txt. Only extract publicly available titles and brief summaries — never reproduce full copyrighted articles. The scraped intelligence is used to INFORM the AI's analysis, not to be reproduced verbatim.

**Caching:** Store scraped intelligence in a new database table (`external_intelligence`) with a TTL of 24 hours. This prevents redundant scraping and provides a fallback if scraping fails during newsletter generation.

import { Resend } from 'resend';

const resend = new Resend(process.env.RESEND_API_KEY);

async function dispatchNewsletter(newsletter: Newsletter): Promise<void> {
  // 1. Get all subscribed users
  const subscribers = await db.query(
    'SELECT email, unsubscribe_token FROM user_emails WHERE newsletter_opt_in = true'
  );

  // 2. Build the HTML email from the newsletter markdown
  const htmlContent = renderNewsletterEmail(newsletter);

  // 3. Send to each subscriber via Resend batch API
  // Resend supports batch sending up to 100 emails per API call
  const batches = chunkArray(subscribers, 100);

  for (const batch of batches) {
    await resend.batch.send(
      batch.map(subscriber => ({
        from: 'AfriEnergy Insights <insights@send.afrienergytracker.io>',
        to: subscriber.email,
        subject: `AfriEnergy Insights — ${newsletter.title}`,
        html: htmlContent.replace('{{UNSUBSCRIBE_URL}}',
          `https://afrienergytracker.io/api/newsletter/unsubscribe?token=${subscriber.unsubscribe_token}`
        ),
      }))
    );
  }

  // 4. Update last_newsletter_sent_at for all recipients
  await db.query(
    'UPDATE user_emails SET last_newsletter_sent_at = NOW() WHERE newsletter_opt_in = true'
  );
}
```
