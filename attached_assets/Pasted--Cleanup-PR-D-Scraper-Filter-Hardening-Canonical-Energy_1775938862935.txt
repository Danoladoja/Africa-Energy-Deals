# Cleanup PR D — Scraper Filter Hardening (Canonical Energy Sector Gate)

## Context

We keep seeing non-energy items slip into the Data Pipeline. This PR hardens the scraper classifier with a **strict sector-match gate** that runs **before** any candidate is inserted into `energy_projects`. Items that fail the gate are rejected, logged with structured telemetry, and never touched again.

---

## Live Audit Findings (2026-04-11)

Before drafting the rules, I audited the live Tracker Admin → Data Pipeline panel at `afrienergytracker.io/admin` (authenticated as ADMINISTRATOR, tab 1387355077). Findings below drive every rule in this PR. **Do not treat these as guesses — they are real titles pulled from the live Rejected/Needs Source/Pending buckets.**

### Bug 1 — World Bank adapter defaults unclassifiable items to "Solar"

The Rejected bucket currently holds ~90 items and **every single one is labeled "Solar"**, even though none of them are solar projects. Real titles observed:

- `Kiira Motors EV Expansion` (electric vehicle manufacturing)
- `Kiira Motors 450-Bus Electric Bus Deal`
- `Spiro Electric Motorcycle Fleet`
- `Nigeria AfDB Economic Governance and Energy Transition Loan` (budget support DPF)
- `Senegal DPF` / `Cote d'Ivoire Private Investment and Productive Jobs DPF1` / `South Africa Infrastructure Modernization and Job Creation DPL`
- `Integrated Social Protection for Resilience and Opportunity`
- `Street-Connected Children and Adolescents Socio-Economic Inclusion`
- `Skills for Employment and Economic Transformation Mozambique`
- `DRC Skills for Economic Transformation`
- `Generating Resilience Opportunities and Welfare for a Thriving Egypt (GROWTH) 2`
- `Morocco Climate & Risk Finance Program`
- `Blended Finance Platform for Resilient Infrastructure`
- `Sudan Food and Nutrition Security`
- `Housing Finance Lands and Sustainable Investments`
- `Kinshasa Urban Transformation and Jobs Program`
- `Support to Transparency in Energy Transition Minerals Development`
- `Second Community Action for Landscape Management`

**Root cause:** The World Bank API adapter (`services/scraper/adapters/worldbank*`) falls back to `technology = 'Solar'` when it cannot classify. This is the single largest source of contamination and is fixed by this PR: the adapter must return `null` sector for unclassifiable items, and the gate must reject any candidate whose sector is `null`.

### Bug 2 — Geothermal projects mislabeled as "Bioenergy"

In the Needs Source bucket, three clearly-geothermal projects are tagged `Bioenergy`:

- `Menengai II Geothermal`
- `Olkaria Expansion`
- `Olkaria IV`

The classifier's keyword matcher is collapsing `geothermal` into the biomass/bioenergy bucket. The sector keyword lists below must make `geothermal` a strong, exclusive match for Geothermal and ensure Biomass keywords do not accidentally swallow it.

### Bug 3 — Sector name drift vs. canonical enum

Live data contains sector labels that are **not** in the proposed canonical enum:

- `Bioenergy` (proposed canonical: `Biomass`)
- `Grid Expansion` (proposed canonical: `Transmission & Distribution`)

**Decision required this PR:** update the extractors and any legacy rows to emit canonical names, rather than adding aliases. Aliases multiply the number of valid strings and make future cleanup harder. Treat the canonical enum as the only allowed output. Legacy rows with drifted names get rewritten in the backfill step.

### Bug 4 — REIPPPP Bid Window 7 duplicated

`REIPPPP Bid Window 7` appears at least 3x across Pending/other buckets. Not a classifier problem per se, but the audit surfaced it — flag in the backfill report for deduplication as a follow-up.

### Bug 5 — Cancelled deals sitting in Pending

`Adani Kenya Electricity Transmission Deal (Cancelled)` is in the Pending bucket tagged as `Grid Expansion`. Not this PR's job to fix status transitions, but the classifier should not block on it; it's flagged here so Daniel can decide to move it manually.

### Failure-mode summary

All the bad items share one of these patterns:

1. **World Bank development policy / budget support loans** — "DPF", "DPL", "development policy financing", "budget support", "economic governance" — these are fiscal instruments, not projects.
2. **Social protection / livelihoods / skills / welfare programs** — "social protection", "skills for employment", "GROWTH", "street-connected", "child welfare", "inclusion project".
3. **Electric vehicles / e-mobility** — "electric vehicle", "electric bus", "motorcycle fleet", "EV manufactur" — demand-side transport, not generation.
4. **Climate / risk finance programs without a named asset** — "climate & risk finance program", "blended finance platform", "carbon market".
5. **General development programs** — "housing finance", "food and nutrition security", "landscape management", "urban transformation".
6. **Transition minerals advocacy** — "transparency in energy transition minerals" — mining governance, not generation.

Every rule in the Hard Rejection Patterns section below is grounded in one of these six patterns.

## Original Observed Failure Modes (kept for context)

1. **Climate / ESG finance with no underlying energy asset** — carbon credit programs, sustainability bonds, green funds, general climate-finance news where no specific energy project is named.
2. **Social / development programming** — youth empowerment, resilience, livelihoods, "just transition" advocacy pieces, community development grants.

## Scope

**In scope:**
- Define a canonical `EnergySector` enum in shared constants. One source of truth.
- Add a `classifyEnergySector(candidate)` function that returns a sector or `null`.
- Insert a gate in the ingest path: if `classifyEnergySector` returns `null`, reject the candidate and log it. Do **not** insert into `energy_projects`.
- Add structured rejection telemetry to `scraper_runs` (counter fields) and emit a line to the run log for each rejection.
- Apply the gate to **every** adapter in `ADAPTER_REGISTRY`.
- Backfill: one-time script to re-classify existing `energy_projects` rows and flag any that don't match the enum. Review-only. No deletes.

**Out of scope:**
- Rewriting adapters.
- Touching the auto-approval confidence thresholds.
- Touching `/discovery` (handled by PR B).
- Touching badges (handled by PR C).
- Creating a rejected-candidates DB table (we chose not to persist rejects to avoid storage growth; telemetry is enough to tune rules).
- Changing the manual URL editing flow.

## Canonical Sector Enum

Add to `lib/shared/src/energy-sectors.ts` (create file if it doesn't exist). Export from the package index.

```ts
// lib/shared/src/energy-sectors.ts

/**
 * Canonical list of energy sectors tracked by AfriEnergy Tracker.
 * THIS IS THE ONLY PLACE new sectors should be added.
 * Adding a sector requires a code change + PR, on purpose.
 */
export const ENERGY_SECTORS = [
  'Solar',
  'Wind',
  'Hydro',
  'Geothermal',
  'Biomass',
  'Nuclear',
  'Oil & Gas',
  'Transmission & Distribution',
  'Battery Storage',
  'Green Hydrogen',
] as const;

export type EnergySector = (typeof ENERGY_SECTORS)[number];

export function isEnergySector(value: string): value is EnergySector {
  return (ENERGY_SECTORS as readonly string[]).includes(value);
}
```

**IMPORTANT:** Before finalizing this enum, run `SELECT DISTINCT technology FROM energy_projects ORDER BY 1;` against the Replit dev DB and reconcile the output against the list above. If a legitimate sector is missing, add it here. If a junk value is present in the DB (e.g. "Climate Finance", "Youth Empowerment"), that is the bug this PR exists to fix — do **not** add it to the enum.

Update the `technology` column type on `energy_projects` if it is currently untyped `text`. Either:
- Migrate the column to use a CHECK constraint against `ENERGY_SECTORS`, or
- Leave the column as `text` but enforce the constraint in application code via Drizzle before insert.

Pick whichever is less risky for existing data. If the DB already has junk values, option 2 is safer (no failing migration) and you can clean up values in the backfill step.

### Hard Rejection Patterns (disqualifiers)

All patterns below are grounded in **real titles** observed in the live Rejected / Needs Source buckets on 2026-04-11 (see Live Audit Findings above).

If the **title** or **description** matches any of these patterns AND does not also explicitly name a generation asset (megawatts, a named power plant, a specific project), reject immediately:

**Development finance / fiscal instruments** (reason: `dev_policy_financing`)
- `/\bDPF\d?\b|\bDPL\d?\b/i` — Development Policy Financing / Loan codes
- `/development policy (financing|loan|operation)/i`
- `/budget support|economic governance|fiscal consolidation/i`
- `/private investment and productive jobs/i`
- `/infrastructure modernization and job creation/i`

**Social programs / welfare** (reason: `social_programming`)
- `/social protection|social inclusion|socio.economic inclusion/i`
- `/skills for (employment|economic transformation)/i`
- `/street.connected children|child welfare|adolescents/i`
- `/resilience opportunities and welfare|GROWTH \d/i`
- `/youth empowerment|just transition|livelihood|community development/i`
- `/food (and|&) nutrition security/i`
- `/landscape management|community action for landscape/i`
- `/housing finance|lands and sustainable investments/i`
- `/urban transformation|jobs program/i`

**E-mobility / demand-side transport** (reason: `e_mobility_not_generation`)
- `/electric vehicle|EV (manufactur|expansion|fleet)|e.mobility/i`
- `/electric bus|bus (deal|fleet)/i`
- `/motorcycle fleet|e.motorcycle/i`
  - *Exception*: if the title also matches `/charging (station|infrastructure|hub)|grid.connected (EV|charging)/i`, do NOT reject — route to `Transmission & Distribution`.

**Climate / ESG finance without a named asset** (reason: `climate_finance_no_asset`)
- `/climate.{0,20}(risk )?finance (program|platform|facility|accelerator)/i`
- `/blended finance platform|resilient infrastructure platform/i`
- `/carbon credit|carbon market|carbon offset/i` when text names no specific project
- `/sustainability bond|green bond/i` unless proceeds are explicitly tied to a named energy project
- `/climate fund|climate finance accelerator/i` as a standalone topic

**Minerals / transition metals advocacy** (reason: `minerals_advocacy`)
- `/transition minerals|critical minerals|transparency in.{0,20}minerals/i`
- `/mining|copper|cobalt|lithium|rare earth/i` unless the text is about a captive power plant serving the mine (look for `/captive (power|plant)|\d+\s*MW.*(mine|mining)/i`)

**MOUs / policy / ceremony** (reason: `mou_no_deal` / `policy_only` / `conference` / `appointment_news`)
- `/mou signed|memorandum of understanding/i` without a specific deal size or asset
- `/policy|regulatory|white paper|feasibility study/i` as the primary topic
- `/conference|summit|workshop|panel discussion|COP\d{2}/i`
- `/appointment|new (ceo|cfo|vice president)|resignation|board (change|reshuffle)/i`

Each rejection sets `rejectionReason` to the code in parentheses. Codes appear in telemetry and make rules tunable from the admin panel.

### Sector Keyword Scoring

For each canonical sector, maintain a keyword list. Score = count of keyword hits in combined title + description. Pick the sector with the highest score. If the highest score is `< 2`, reject with reason `"no_sector_signal"`. If two sectors tie above threshold, accept the one that matches `extractedTechnology` when present, otherwise reject with reason `"ambiguous_sector"`.

Suggested starter keyword lists:

- **Solar**: solar, PV, photovoltaic, concentrated solar, CSP, solar farm, solar plant, solar park
- **Wind**: wind farm, wind turbine, wind power, offshore wind, onshore wind
- **Hydro**: hydropower, hydroelectric, dam, run-of-river, pumped storage
- **Geothermal**: geothermal, steam field, wellhead, Olkaria, Menengai (named Kenyan geothermal fields — high-confidence tokens)
- **Biomass**: biomass, biogas, biofuel, waste-to-energy, bagasse
- **Nuclear**: nuclear reactor, nuclear power plant, SMR, small modular reactor
- **Oil & Gas**: LNG, gas field, oil field, upstream, FPSO, refinery, pipeline, crude oil (reject pure pipeline-infra if no deal attached — see disqualifiers)
- **Transmission & Distribution**: transmission line, grid extension, substation, interconnector, HVDC, grid expansion
- **Battery Storage**: battery storage, BESS, lithium-ion storage, grid-scale battery
- **Green Hydrogen**: green hydrogen, electrolyzer, ammonia (green), H2 project

**Geothermal vs. Biomass collision fix (Bug 2):** The classifier must match `geothermal` as an **exclusive** Geothermal token. If the text contains `geothermal`, score Biomass at zero for that candidate regardless of other matches. This prevents Olkaria/Menengai from being tagged "Bioenergy" as they currently are. Add a test case for `"Menengai II Geothermal"` asserting `Geothermal`, not `Biomass`.

Keywords live in the same file so they're reviewable together. Do not move them into the DB.

### Confidence Score

Return `confidence = min(1, keywordHits / 4)`. Adapters that wrap this classifier can combine this signal with their own extraction confidence. Existing auto-approval thresholds (≥0.85 → approved, 0.65–0.84 → pending) continue to apply downstream.

## Auto-Purge of Rejected Items (No Clutter, No DB Bloat)

**Rejected items must never occupy space in `energy_projects`.** The gate's job is to stop them at the door, not to stage them in a "Rejected" bucket the operator has to sweep later. The current ~90-item Rejected bucket is itself the clutter this PR exists to eliminate.

### Rules

1. **Pre-insert rejection (primary path):** When `classifyEnergySector` returns `null`, the candidate is dropped immediately. **No row is written to `energy_projects`.** The only record of it is:
   - An increment to `scraper_runs.rejected_non_energy_count` for the current run
   - An entry (title, URL, reason, adapter) in `scraper_runs.rejection_log` (bounded at 100 per run)
   - A structured log line (`scraper.rejected` with the same fields) for Railway logs

   That's it. No row in `energy_projects`, no row in any "rejected" table, no staging area.

2. **Existing Rejected-bucket purge (one-time cleanup):** The ~90 World Bank adapter rows currently sitting in the admin's "Rejected" bucket must be hard-deleted as part of this PR's backfill script. Add a third allowed mutation to `backfill-sector-classification.ts`:

   ```sql
   DELETE FROM energy_projects
   WHERE status = 'rejected'
     AND classifyEnergySector(projectName || ' ' || COALESCE(description, '')) IS NULL;
   ```

   (Use a TypeScript loop, not raw SQL, so the same classifier function runs.) Log every deletion to the backfill report (title + id + reason) so Daniel has an auditable record of what was removed. Print a summary count at the end.

3. **Ongoing auto-purge of stale Rejected/Needs-Source rows:** Add a lightweight cron or Railway scheduled job (reuse whatever pattern the project already has — do not introduce a new scheduler) that runs once per day and:
   - Deletes any `energy_projects` row with `status IN ('rejected')` older than 7 days.
   - Deletes any `energy_projects` row with `status = 'needs_source'` older than 30 days with no reviewer action.
   - Logs counts to `scraper_runs` or a new `purge_runs` table (single row per run, cheap).

   Configurable retention windows live in `lib/shared/src/constants.ts`:
   ```ts
   export const PURGE_RETENTION_DAYS = {
     rejected: 7,
     needsSource: 30,
   } as const;
   ```

4. **Telemetry-log trimming:** `scraper_runs.rejection_log` is capped at 100 entries per run, and `scraper_runs` itself is purged to the last 90 days by the same daily job. Rejection telemetry is for **trend watching, not forensic retention**.

5. **Admin UI:** The Data Pipeline tab's "Rejected" bucket becomes a **live view of the last 7 days only**, sourced from `scraper_runs.rejection_log` rather than `energy_projects`. The bucket should never again fill up with rows that need manual cleanup.

### Rationale

The current admin dashboard treats the Rejected bucket as a permanent ledger. That's how we ended up with 90 mislabeled "Solar" rows cluttering both the admin UI and the database. The correct model is: **rejection is a runtime event, not a stored entity**. Keep a short tail of telemetry for tuning, delete everything else.

### Safety

- All deletions are gated on `classifyEnergySector` returning `null` — no row with a real sector match is ever auto-deleted.
- The daily purge runs in a single transaction per batch so a crash mid-run leaves the DB consistent.
- A one-line feature flag `PURGE_ENABLED=true` (env var) lets Daniel disable the job instantly if anything looks wrong in the first 48 hours after deploy.

## Gate Integration

Every adapter funnel in `artifacts/api-server/src/services/scraper/` must call `classifyEnergySector` before staging the candidate for insertion. The pipeline should look like:

```
fetch → parse → extract candidate → classifyEnergySector → [gate] → stage → insert
```

If `classifyEnergySector` returns `sector: null`:
1. **Do not** create a row in `energy_projects`.
2. **Do not** create a row in any review queue.
3. Increment `scraper_runs.rejected_non_energy_count` for the current run.
4. Append to `scraper_runs.rejection_log` (JSON array) a record of:
   ```json
   {
     "title": "...",
     "sourceUrl": "...",
     "reason": "carbon_credit_no_asset",
     "matchedKeywords": [],
     "adapter": "dfi:afdb",
     "rejectedAt": "2026-04-11T..."
   }
   ```
5. Emit a structured log line: `sector_gate.reject` with the same fields.

If `classifyEnergySector` returns a sector, store it on the staged candidate as the authoritative `technology` value and proceed with the existing flow.

### New columns on `scraper_runs`

Add to `lib/db/src/schema/scraper.ts` and to `migrate.ts`:

```sql
ALTER TABLE scraper_runs
  ADD COLUMN IF NOT EXISTS rejected_non_energy_count INTEGER DEFAULT 0,
  ADD COLUMN IF NOT EXISTS rejection_log JSONB DEFAULT '[]'::jsonb;
```

Rejection log is **bounded**: append only up to 100 entries per run to prevent unbounded growth. After 100, increment the counter but drop the sample. This is telemetry, not a permanent rejection store.

## Backfill (One-Time, Review-Only)

Create `artifacts/api-server/scripts/backfill-sector-classification.ts`:

1. `SELECT id, projectName, description, technology FROM energy_projects;`
2. **Sector name normalization (the one allowed mutation):**
   - `UPDATE energy_projects SET technology = 'Biomass' WHERE technology = 'Bioenergy';`
   - `UPDATE energy_projects SET technology = 'Transmission & Distribution' WHERE technology = 'Grid Expansion';`
   - Log counts of both rewrites to the report.
3. For each remaining row, run it through `classifyEnergySector`.
4. If the result is `null` OR disagrees with the existing `technology` value, write the row to `/tmp/sector-backfill-report.json` with the classifier's opinion.
5. Separately, detect duplicates by `projectName` (case-insensitive) and append a `duplicates` section to the report (surfaces the REIPPPP Bid Window 7 triple-count observed in the audit).
6. **Do not mutate any rows beyond the two name-normalization UPDATEs above.** Everything else is a review-only audit pass.
7. At the end, print a summary: total rows, name-drift fixes applied, rows flagged for review, duplicate groups found, breakdown by rejection reason.

After the script runs, hand the report back to Daniel for review. Any rows he confirms as bad will be cleaned up manually via the admin Projects table UI (or a follow-up PR if volume is high). Expected to flag ~90 rows currently sitting in the Rejected bucket that are mislabeled "Solar" — these are the World Bank adapter defaults.
