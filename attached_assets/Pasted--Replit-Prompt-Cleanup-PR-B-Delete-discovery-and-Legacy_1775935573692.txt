**Repo:** AfriEnergy Tracker Replit monorepo
**Priority:** Boring cleanup. Ship after Cleanup PR A (migration hotfix) is merged.
**Context:** The `RECOVERY_AUDIT.md` confirmed that `/discovery` is still live: the page component, the route registration, the lazy import, and 14 legacy `/api/scraper/*` flat-function routes all remain. The Admin Dashboard Pipeline section (`/admin?section=pipeline`) is the canonical replacement. This PR deletes the old surface cleanly.

**Scope is deliberately narrow.** This PR is a deletion, not a feature. If you find yourself adding functionality anywhere, stop.

---

## GOAL

1. Remove the `/discovery` page, route, and lazy import.
2. Remove the 14 legacy `/api/scraper/*` flat-function routes in `routes/scraper.ts`.
3. Add a one-time redirect from `/discovery` → `/admin?section=pipeline` so stale bookmarks don't 404.
4. Verify nothing else in the codebase still imports from the deleted files.

## SCOPE — WHAT TO DELETE

### Frontend

- `artifacts/energy-tracker/src/pages/discovery.tsx` — delete the file entirely.
- `App.tsx:27` — remove the lazy import `const DiscoveryPage = lazy(() => import("@/pages/discovery"))`.
- `App.tsx:213` — remove the `<Route path="/discovery">` registration.
- Any import of `DiscoveryPage` elsewhere — remove.
- Any reference to `/discovery` in the nav, sidebar, breadcrumbs, or links — remove. The nav item was already removed from `adminNavItems`, but search for any remaining references and clean them up.

### Backend

- `artifacts/api-server/src/routes/scraper.ts` — this file has 14 legacy endpoints. Confirm which (if any) are still consumed by the Admin Dashboard Pipeline section before deleting. The audit says `/admin?section=pipeline` uses the new adapter system via `routes/adapters.ts`, so the legacy flat-function `scraper.ts` routes should be unreferenced. **Before deleting, grep for every endpoint path and every exported function name** to confirm nothing else consumes them. If anything does, stop and ask.
- If `scraper.ts` contains helper functions used by the new adapter system, extract them to a shared location before deleting the routes — do not delete code that's still called.
- Any tests targeting the deleted endpoints — delete.

### Redirect

Add a one-line redirect rule in `App.tsx` (or wherever routing is centralised) so `/discovery` resolves to `/admin?section=pipeline` with a 301/302. Do this **before** deleting the page file, so the redirect tests pass. Then the page file can be removed because the route no longer points at it.

Alternatively, if Wouter (or whatever router is in use) supports a redirect component, use that. The goal is a permanent redirect, not a page that rerenders.
